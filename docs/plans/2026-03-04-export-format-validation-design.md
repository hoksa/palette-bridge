# Export Format Validation: Palette Bridge vs Material Theme Builder

## Context

Evaluated Palette Bridge's Kotlin and Material JSON exports against Material Theme Builder (MTB) web tool exports to validate that our team can use PB exports as drop-in replacements.

Reference exports compared:
- MTB Jetpack Compose zip: `Color.kt`, `Theme.kt`, `Type.kt`
- MTB Material JSON: `material-theme.json`

## Kotlin Export: Fully Compatible

PB's Kotlin export (`Color.kt` + `Theme.kt`) is **structurally identical** to MTB.

### Color.kt

| Property | MTB | Palette Bridge | Match |
|----------|-----|---------------|-------|
| Roles per variant | 35 | 35 | Yes |
| Excluded roles | shadow, surfaceTint, 12 fixed | Same | Yes |
| Naming convention | `{role}{Light}`, `{role}{DarkMediumContrast}` | Same | Yes |
| Color format | `Color(0xFFRRGGBB)` | Same | Yes |
| Variants | 6 (Light, Dark, LightMediumContrast, DarkMediumContrast, LightHighContrast, DarkHighContrast) | Same | Yes |

### Theme.kt

| Property | MTB | Palette Bridge | Match |
|----------|-----|---------------|-------|
| Scheme count | 6 | 6 | Yes |
| Scheme names | `lightScheme`, `darkScheme`, `mediumContrastLightColorScheme`, etc. | Same | Yes |
| Constructors | `lightColorScheme()` / `darkColorScheme()` | Same | Yes |
| Role parameters | 35 per scheme | Same | Yes |

### Not generated by PB (not needed)

MTB also exports `AppTheme` composable, `ColorFamily` data class, `unspecified_scheme`, and `Type.kt`. These are boilerplate already present in our Android codebase — PB correctly omits them since only `Color.kt` and scheme declarations need replacing.

### Verdict

PB's Kotlin export can be dropped directly into the Android project as a replacement for the existing MTB-generated files. No structural changes needed.

## Material JSON Export: Compatible with Documented Differences

PB's Material JSON export matches MTB's structure. Two intentional differences exist.

### Structural match

| Property | MTB | Palette Bridge | Match |
|----------|-----|---------------|-------|
| Top-level keys | description, seed, coreColors, extendedColors, schemes, palettes | Same | Yes |
| Scheme count | 6 | 6 | Yes |
| Scheme names | light, dark, light-medium-contrast, etc. | Same | Yes |
| Roles per scheme | 49 | 49 | Yes |
| Role order | MTB insertion order | Same | Yes |
| Hex format | Uppercase `#RRGGBB` | Same | Yes |
| Palette families | primary, secondary, tertiary, neutral, neutral-variant | Same | Yes |
| Tones per palette | 18 (0, 5, 10, 15, 20, 25, 30, 35, 40, 50, 60, 70, 80, 90, 95, 98, 99, 100) | Same | Yes |

### Difference 1: `coreColors` includes all palette families

**MTB:** Only includes the seed color(s) that were explicitly customized. For a single-seed theme, this is just `{"primary": "#hex"}`.

**PB:** Includes all 5 families (primary, secondary, tertiary, neutral, neutralVariant) because PB's model has all palettes as explicit user input rather than derived from a single seed.

**Impact:** Low. This is a model difference, not a bug. PB provides more information than MTB. Downstream consumers that read `coreColors` will find all families present rather than just the seed.

### Difference 2: `neutral-variant` palette duplicates `neutral`

**MTB:** Generates distinct tonal palettes for neutral (~2 chroma) and neutral-variant (~4 chroma) using HCT color science. These produce visibly different tones.

**PB:** Uses a single user-provided neutral Tailwind palette for both. The `palettes` section shows identical values for `neutral` and `neutral-variant`.

**Impact:** Medium, but limited to the `palettes` reference section. The `schemes` section (light, dark, contrast variants) contains correctly resolved hex values for all roles. Downstream tools that read `schemes` directly are unaffected. Tools that regenerate schemes from `palettes` would not distinguish neutral from neutral-variant.

**Future mitigation (if needed):** Auto-derive a neutral-variant tonal palette by adding slight chroma offset to the neutral palette in OKLCH space. This would be Approach B from the evaluation.

## Recommendations

1. **Kotlin:** Ship as-is. No changes.
2. **Material JSON:** Ship as-is. The `schemes` section is correct; the `palettes` duplication is a known limitation.
3. **If neutral-variant distinction becomes important:** Implement auto-derivation (Approach B) as a follow-up.
